## Objetivo
Hacer funcionar `=NEVEN.r("1+1")` en Excel, como primer paso hacia la paridad funcional con BERT Toolkit original.

---

## Contexto Inicial

- Proyecto: NEVEN — Add-in de Excel para ejecutar R y Julia desde celdas
- Basado en: BERT Toolkit (sdllc/Basic-Excel-R-Toolkit)
- Problema reportado: Al cargar el Addin en Excel, no se muestra ninguna función relacionada con R o Julia
- Entorno: Windows 11, Excel 365 64-bit, R 4.4.1, Julia 1.12.6, VS 2022

---

## Diagnóstico Inicial (Estado de las Cosas)

### Problemas identificados:
1. `ControlR.exe` no compilaba (R_Environment.cpp usaba `xloper12` de Excel en proceso hijo)
2. `neven-languages.json` no pasaba `-r $HOME` a ControlR.exe
3. `MessageBoxA()` de debug bloqueaban la ejecución
4. Faltaba `startup/startup.r`
5. `ValidFile()` usaba `find()` en lugar de comparar extensiones exactas (`.jl` matcheaba a R)
6. Las constantes del Excel SDK estaban mal: `xlGetName=107` debería ser `0x4019`
7. La firma de `MdCallBack12` estaba invertida en `xlcall_stubs.cc`
8. `Pipe::Start(wait=true)` esperaba en `handle_` en lugar de `read_io_.hEvent`

- `ControlR/CMakeLists.txt`: Cambiado de `add_library` a `add_executable`, auto-detección de R_HOME, exclusión de `R_Environment.cpp`
- `Addin/CMakeLists.txt`: Copia ControlR.exe, ControlJulia.exe, startup scripts y ejemplos a `Dist/`
- Script `scripts/rebuild-r-libs.ps1`: Regenera R64.lib y RGraphApp64.lib desde R 4.4.1
- Script `scripts/build-all.ps1`: Build completo automatizado

### Correcciones de Código
1. **`xlcall_stubs.cc`**: Corregida firma de `MdCallBack12`:
   - Incorrecta: `(xlfn, operRes, count, opers[])`
   - Correcta: `(xlfn, count, opers[], operRes)`

2. **`Include/XLCALL.h`**: Corregidas constantes del Excel SDK:
   - `xlGetName`: 107 --> `0x4019` (16409)
   - `xlCoerce`: 2 --> `0x4002` (16386)
   - `xlFree`: 161 --> `0x4000`

3. **`NEVEN.cc::xlAutoOpen`**: Busca `MdCallBack12` por nombre en Excel.exe

4. **`excel_api_functions.cc::RegisterBasicFunctions`**: 
   - Usa `xlGetName` para obtener el nombre del DLL
   - Fallback con `GetModuleFileNameW` + corrección de extensión `.dll` --> `.xll`
   - Eliminados `MessageBoxA` de debug

5. **`neven-languages.json`**: Agregado `-r "$HOME"` como `command_arguments` para R, y `prepend_path` para ambos lenguajes

6. **`language_service.cc::ValidFile`**: Corregido para comparar extensión exacta en lugar de substring

7. **`startup/startup.r`**: Creado (faltaba completamente)

8. **`NEVEN.cc::Init`**: Log a archivo `C:\NEVEN\neven.log`, crea directorio de funciones del usuario

9. **`Common/pipe.cc::Pipe::Start`**: Corregido `WaitForSingleObject(handle_)` --> `WaitForSingleObject(read_io_.hEvent)`

### Configuración
- `neven-config.json`: `openConsole: false`, `useJobObject: true`
- Log path: `C:\NEVEN\neven.log`

## Problema Actual: ControlR.exe Crashea

### Síntoma
ControlR.exe se lanza desde Excel pero crashea antes de que R entre en su loop principal. El crash ocurre dentro de `RLoop()` en `rinterface_win.cc`, específicamente antes de `R_SetParams()`.

### Evidencia del log `C:\NEVEN\controlr.log`
```
ControlR starting...
R version: 4.4.1
Starting RLoop with rhome=C:\Program Files\R\R-4.4.1
```
El log termina aquí — el crash ocurre en `R_setStartTime()`, `R_DefParams()`, o al asignar los parámetros del struct `Rstart`.

### Diferencia entre manual y desde Excel
- **Manual** (desde Developer Command Prompt): ControlR.exe funciona perfectamente
- **Desde Excel** (lanzado por el XLL): Crashea a los 4 segundos

### Hipótesis
El crash puede ocurrir en `GA_initapp(0, 0)` o `readconsolecfg()` — funciones de `RGraphApp.dll` que requieren una ventana gráfica o consola. Cuando Excel lanza ControlR.exe con `CREATE_NO_WINDOW`, estas funciones pueden fallar.

### Cambio en progreso
- Usando headers reales de R 4.4.1 para ControlR.exe con `target_include_directories(BEFORE)`
- Creado `ControlR/include/R_ext/Complex.h` compatible con MSVC (evita `double _Complex` de C99)
- Corregida firma de `R_ReadConsole` (`char*` --> `unsigned char*` en R 4.4.1)
- Cambiado `CharacterMode = LinkDLL` (sin GUI)
- Deshabilitado callback thread y FileWatchService temporalmente
- Simplificado `Call()` para esperar solo en main pipe con timeout 30s

## Problemas Secundarios Identificados

### ControlJulia.exe
- Exit code: `-1073741511` = `0xC0000139` = `STATUS_ENTRYPOINT_NOT_FOUND`
- Julia 1.12.6 puede tener cambios de API incompatibles con el código actual
- Pendiente de investigación

### Callback Pipe
- El callback pipe (para callbacks de R --> Excel como `plot()`) se rompe 3 segundos después de conectar
- Causa: race condition entre el callback thread del XLL y el startup de R en ControlR
- Solución temporal: callback thread deshabilitado
- Impacto: funciones básicas de R funcionarán, pero `plot()` y COM automation no

### ERR getting dll name
- `xlGetName` retorna error 2 (`xlretInvXloper`) en algunos contextos
- El fallback con `GetModuleFileNameW` funciona correctamente
- No impacta la funcionalidad

## Comandos de Build

```powershell
# Desde Developer Command Prompt for VS 2022, en F:\ANTIGRAVITY\2026\NEVEN\NEVEN\Build

# Build completo
cmake .. -A x64 -DR_HOME="C:\Program Files\R\R-4.4.1"
cmake --build . --config Release

# Solo el XLL (más rápido)
cmake --build . --config Release --target NEVEN_Core -- /p:ForceRebuild=true
cmake --build . --config Release --target NEVEN_XLL

# Solo ControlR
cmake --build . --config Release --target ControlR

# Deploy
taskkill /f /im excel.exe 2>nul
taskkill /f /im ControlR.exe 2>nul
copy /Y Dist\NEVEN64.xll C:\NEVEN\NEVEN64.xll
copy /Y Dist\ControlR.exe C:\NEVEN\ControlR.exe

# Ver logs
type C:\NEVEN\neven.log
type C:\NEVEN\controlr.log
```
